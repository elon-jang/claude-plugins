"99%의 작업은 모델 안에서 이루어진다."
Browser Use 팀이 에이전트를 다 뜯어고치고 나서 깨달은 겁니다.

그들의 첫 에이전트는 수천 줄의 추상화로 이루어져 있었습니다.
작동은 했습니다. 뭔가를 바꾸려고 하기 전까지는요.
모든 실험이 프레임워크와 싸워야 했습니다.

결국 전부 버렸습니다.
에이전트가 실패한 건 모델이 멍청해서가 아니었습니다.
프레임워크가 모델을 방해하고 있었습니다.

1. Bitter Lesson

ML 연구의 교훈은 명확합니다.
"계산을 활용하는 일반적인 방법이 수작업 지식을 매번 이긴다."

모델 주위에 "똑똑한" 래퍼를 추가할 때마다

- 계획 모듈
- 검증 레이어
- 출력 파서

당신이 생각하는 "모델이 해야 할 것"을 인코딩하는 겁니다.
하지만 모델은 수백만 개의 예제로 훈련됐습니다.
당신보다 더 많은 패턴을 봤습니다.

추상화는 제약이 됩니다.
모델이 배운 것을 사용하지 못하게 막습니다.

2. 진짜 필요한 것

Browser Use 팀의 결론은 단순했습니다.

"에이전트는 그저 메시지의 for-loop일 뿐이다.
유일한 상태는: 모델이 도구 호출을 멈출 때까지 계속 진행하는 것."

에이전트에게 필요한 건 세 가지뿐입니다.

- 도구
- 컨텍스트
- for-loop

Claude Code가 AppleScript를 직접 작성할 수 있습니다.
문자 보내달라고 하면? SMS 도구가 필요 없습니다.
그냥 메시지 앱을 AppleScript로 제어합니다.

모든 사용 사례를 예상할 필요 없습니다.
모델이 이미 알고 있습니다.

3. 역전의 발상

기존 접근: 가능한 행동을 미리 정의 → 모델에게 제공
새로운 접근: 모델은 거의 모든 것을 할 수 있다 → 그다음 제한

LLM에게 최대한의 자유를 주고, eval 기반으로 제한하세요.

원칙적으로 모든 것이 가능하기만 하면,
LLM은 즉석에서 스스로를 고치는 데 극도로 뛰어납니다.
한 접근이 실패하면 우회합니다.
도구가 고장나면 다른 경로를 찾습니다.

4. 직접 확인해봤습니다

Claude Code 내부를 분석해봤습니다.
도구는 15개 정도. Bash, Read, Write, Edit, Task 등.
복잡한 계획 모듈이나 검증 레이어가 없습니다.

핵심은 정말 단순했습니다.
도구를 주고, 컨텍스트를 주고, 모델이 알아서 하게 둔다.
도구 호출이 없으면 종료.

"적게 만들수록 더 잘 작동한다"가 코드에 그대로 담겨 있었습니다.

모든 추상화는 부채입니다.
모든 "헬퍼"는 실패 지점입니다.

모델이 좋아졌습니다. 정말 좋아졌습니다.
RL로 computer use, 코딩, 브라우징을 학습했습니다.
당신의 가드레일이 필요 없습니다.
