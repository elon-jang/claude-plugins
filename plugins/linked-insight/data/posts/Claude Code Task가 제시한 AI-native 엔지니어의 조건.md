Claude Code 의 Task 가 제시한 AI-native 엔지니어의 조건

지난 주 Claude Code 의 Todo 가 Task 로 이름이 바뀌었어요. 단순 명칭 변경처럼 보이지만, 이건 완전히 다른 시스템의 시작이에요.

Todo 는 Claude 가 혼자 기억해야 할 목록이었어요. Task 는 여러 에이전트가 공유하는 작업 단위예요. 이 차이가 AI 코딩 도구의 패러다임을 바꿔요.

즉, AI Swarm 에 필요한 새로운 추상화 단위가 등장한 것입니다.

1️⃣ 핵심은 '위임'이에요, '자동화'가 아니라

기존 Claude Code 는 단일 두뇌였어요. 복잡한 작업을 맡기면 중간에 앞서 했던 걸 잊어버리고, 결국 60% 쯤에서 다시 시작하는 일이 반복됐죠.

새 Task 시스템은 구조가 완전히 달라요:

• 여러분은 팀 리더에게 말을 걸어요
• 리더는 직접 코드를 짜지 않아요. 계획하고, 위임하고, 종합해요
• 승인하면 전문가 팀이 생성되어 병렬로 일해요

2️⃣ 의존성 그래프가 진짜 무기예요

Task 간 의존성(blockedBy)이 핵심이에요. 작업 3번은 1번과 2번이 완료될 때까지 시작조차 못해요.

왜 이게 중요할까요?

이전에는 Claude 가 머릿속에 전체 계획을 들고 있어야 했어요. 맥락이 길어지면 자연스럽게 잊어버렸죠.

이제 계획 자체가 외부에 구조화돼요. 맥락이 압축되거나 에이전트가 교체돼도 계획은 살아남아요.

3️⃣ 병렬 처리는 '공짜로' 따라와요

7~10개 작업을 맡기면 이제 순차 처리가 아니에요. 의존성이 없는 건 동시에 돌아가요. Haiku 로 빠른 검색, Sonnet 으로 구현, Opus 로 복잡한 판단까지 자동 배분돼요.

4️⃣ 사용자가 집중해야 할 건 '오케스트레이션'이에요

Swarm 문서를 보면 패턴이 명확해요:

• Parallel Specialists: 전문가 여러 명이 동시에 리뷰
• Pipeline: 연구 → 계획 → 구현 → 테스트 순차 진행
• Self-Organizing Swarm: 작업 풀에서 알아서 가져가는 구조

코드를 짜는 게 아니라, 어떤 에이전트가 어떤 순서로 뭘 하게 할지 설계하는 게 일이 돼요.

5️⃣ Swarm 효율을 극대화하는 핵심은 '작업 설계'예요

Swarm 을 잘 쓰려면 세 가지 접근이 있어요:

• 작업 쪼개기: 병렬화율은 높아지지만 에이전트 간 통신 오버헤드가 늘어요
• 역할 분리: 전문성은 올라가지만 특정 에이전트에 병목이 생길 수 있어요
• 의존성 설계: 뭘 먼저 끝내야 다음이 막힘 없이 도는지 구조화하는 방식이에요

(제가 실험해보니 세 번째가 가장 주요하게 영향을 주었어요)

결국 핵심은 코드가 아니라 작업 흐름의 구조를 짜는 능력이에요. 이걸 저는 개인적으로 '의존성 토폴로지 설계'라고 부르는데, 이게 Swarm 시대의 진짜 실력이에요.

코드를 짜는 시대에서 시스템을 설계하는 시대로, 이제는 일하는 방식 자체를 설계하는 시대로 넘어가고 있어요.
